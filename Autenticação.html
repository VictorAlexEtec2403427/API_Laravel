🔐 Protegendo a API Laravel + SQLite com JWT

✏️ Material didático para Ensino Médio e Técnico — Continuando o projeto da aula anterior
💡 Objetivo da aula: Entender o que é JWT, instalar o pacote tymon/jwt-auth, criar rotas de login/registro, proteger as rotas de produtos e testar tudo no Postman enviando o token como um "crachá digital".
✅ Pré-requisitos: Ter concluído a aula anterior (API de produtos funcionando), PHP >=8.1, Composer, Postman instalado.

🔰 1. O que é JWT? (Analogia do Crachá)

Imagine que sua escola tem uma portaria. Na primeira aula, você mostra seu RG e a secretária te dá um CRACHÁ com seu nome, turma e horário de validade. Durante o dia, você só precisa mostrar o crachá para entrar nas salas — não precisa mostrar o RG de novo.

O JWT (JSON Web Token) funciona exatamente assim:

  • 🔹 register/login = você mostra o RG e ganha um crachá (token).
  • 🔹 token = um texto longo (ex.: eyJhbGciOiJIUzI1...) que carrega seus dados.
  • 🔹 Bearer Token = é o jeito de "pendurar o crachá" em cada requisição.
  • 🔹 expiração = o crachá tem hora para vencer (por padrão 60 minutos).
Vantagens: sem sessão no servidor, funciona em apps mobile, APIs e SPAs.

🛠️ 2. Instalando o pacote tymon/jwt-auth

No terminal, dentro da pasta do projeto:

php composer.phar require tymon/jwt-auth:^2.1

Depois, publique o arquivo de configuração:

php artisan vendor:publish --provider="Tymon\JWTAuth\Providers\LaravelServiceProvider"

E gere a chave secreta (é ela que "assina" o crachá — guarde bem!):

php artisan jwt:secret
✅ Isso cria uma linha JWT_SECRET=xxxxx no seu .env.

🧬 3. Preparando o Model User para JWT

Abra app/Models/User.php e faça duas mudanças:

3.1 Implementar a interface JWTSubject

use Tymon\JWTAuth\Contracts\JWTSubject;

class User extends Authenticatable implements JWTSubject

3.2 Adicionar os dois métodos obrigatórios

public function getJWTIdentifier()
{
    return $this->getKey(); // retorna o id do usuário
}

public function getJWTCustomClaims()
{
    return []; // aqui você pode adicionar dados extras no token
}

3.3 Garantir que $fillable tenha os campos certos

protected $fillable = ['name', 'email', 'password'];
protected $hidden = ['password', 'remember_token'];
📌 Dica do professor: o password PRECISA estar em $hidden para nunca vazar no JSON.

🔐 4. Criando o AuthController

Vamos gerar o controller responsável por login, registro e logout:

php artisan make:controller Api/AuthController

Conteúdo completo de app/Http/Controllers/Api/AuthController.php:

<?php
namespace App\Http\Controllers\Api;

use App\Models\User;
use Illuminate\Http\Request;
use App\Http\Controllers\Controller;
use Illuminate\Support\Facades\Hash;
use Tymon\JWTAuth\Facades\JWTAuth;

class AuthController extends Controller
{
    // Registrar novo usuário
    public function register(Request $request)
    {
        $request->validate([
            'name'     => 'required|string|max:100',
            'email'    => 'required|email|unique:users',
            'password' => 'required|string|min:6|confirmed',
        ]);

        $user = User::create([
            'name'     => $request->name,
            'email'    => $request->email,
            'password' => Hash::make($request->password),
        ]);

        return response()->json([
            'message' => 'Usuário criado com sucesso',
            'user'    => $user
        ], 201);
    }

    // Login → gera o token (o "crachá")
    public function login(Request $request)
    {
        $credentials = $request->only('email', 'password');

        if (!$token = JWTAuth::attempt($credentials)) {
            return response()->json(['message' => 'E-mail ou senha inválidos'], 401);
        }

        return $this->respondWithToken($token);
    }

    // Dados do usuário autenticado
    public function me()
    {
        return response()->json(auth('api')->user());
    }

    // Logout → invalida o token
    public function logout()
    {
        auth('api')->logout();
        return response()->json(['message' => 'Logout realizado com sucesso']);
    }

    // Refresh → gera um novo token (renova o crachá)
    public function refresh()
    {
        return $this->respondWithToken(auth('api')->refresh());
    }

    // Helper: padroniza a resposta do token
    protected function respondWithToken($token)
    {
        return response()->json([
            'access_token' => $token,
            'token_type'   => 'bearer',
            'expires_in'   => auth('api')->factory()->getTTL() * 60
        ]);
    }
}

🧭 5. Definindo as rotas (públicas vs. protegidas)

Abra routes/api.php e substitua tudo por:

<?php
use Illuminate\Support\Facades\Route;
use App\Http\Controllers\Api\AuthController;
use App\Http\Controllers\Api\ProdutoController;

// 🔓 ROTAS PÚBLICAS (qualquer um pode acessar)
Route::post('/register', [AuthController::class, 'register']);
Route::post('/login',    [AuthController::class, 'login']);

// 🔒 ROTAS PROTEGIDAS (precisa do "crachá" JWT)
Route::middleware('auth:api')->group(function () {
    Route::post('/logout',  [AuthController::class, 'logout']);
    Route::post('/refresh', [AuthController::class, 'refresh']);
    Route::get('/me',       [AuthController::class, 'me']);

    Route::apiResource('produtos', ProdutoController::class);
});
📌 O que mudou?
  • /register e /login continuam abertas.
  • Tudo que está dentro do middleware('auth:api') exige token.
  • Se tentar acessar /api/produtos sem token → recebe 401 Unauthorized.

⚙️ 6. Ajuste importante no config/auth.php

Abra config/auth.php e altere o guard "api":

'guards' => [
    'api' => [
        'driver'   => 'jwt',     // ← era 'token', agora é 'jwt'
        'provider' => 'users',
    ],
],
⚠️ Atenção: Sem isso, o Laravel não sabe que deve usar JWT nesse guard.

php artisan serve

A API continua em http://127.0.0.1:8000, mas agora com portaria! 🚪

🧪 8. Testando no Postman — fluxo completo com token

8.1 Registrar um usuário (POST) — PÚBLICO

Método: POST
URL: http://127.0.0.1:8000/api/register
Body (JSON):

{
    "name": "Maria Silva",
    "email": "maria@escola.com",
    "password": "123456",
    "password_confirmation": "123456"
}
✅ Resposta: 201 Created + dados do usuário.

8.2 Fazer login (POST) — PÚBLICO

Método: POST
URL: http://127.0.0.1:8000/api/login
Body (JSON):

{
    "email": "maria@escola.com",
    "password": "123456"
}

✅ Resposta:

{
    "access_token": "eyJ0eXAiOiJKV1QiLCJhbGci...",
    "token_type": "bearer",
    "expires_in": 3600
}
📌 COPIE esse access_token — ele é seu crachá!

8.3 Configurar o Postman para usar o token

Opção A (recomendada): aba Authorization → tipo "Bearer Token" → cole o token.

Opção B: aba Headers → adicione:

Authorization: Bearer eyJ0eXAiOiJKV1QiLCJhbGci...

8.4 Listar produtos (GET) — PROTEGIDO

Método: GET
URL: http://127.0.0.1:8000/api/produtos

✅ Com token: retorna a lista.
❌ Sem token: retorna {"message": "Unauthenticated."} com status 401.

8.5 Criar produto (POST) — PROTEGIDO

Mesmo JSON da aula anterior, mas AGORA com o token configurado.
✅ Funciona normalmente.

8.6 Ver quem está logado (GET)

Método: GET
URL: http://127.0.0.1:8000/api/me
✅ Retorna os dados da Maria (usuário do token).

8.7 Renovar o token (POST)

Método: POST
URL: http://127.0.0.1:8000/api/refresh
✅ Retorna um NOVO access_token (crachá renovado).

8.8 Fazer logout (POST)

Método: POST
URL: http://127.0.0.1:8000/api/logout
✅ Token invalidado. Se tentar usar de novo → 401.

📊 9. Resumo do que aprendemos

Conceito Como aplicamos
JWT Token assinado que substitui sessão
tymon/jwt-auth Pacote que implementa JWT no Laravel
JWTSubject Interface que o User implementa para "ganhar crachá"
AuthController register, login, logout, refresh, me
Middleware auth:api Protege rotas — exige token válido
Bearer Token Forma de enviar o token no cabeçalho HTTP
Guard 'jwt' Configuração em config/auth.php

🔧 10. Possíveis problemas e soluções

  • "Class JWTAuth not found" → rode composer require tymon/jwt-auth e verifique o namespace.
  • "Could not create token: Wrong number of segments" → rode php artisan jwt:secret.
  • 401 em todas as rotas → confira se o guard 'api' está com driver 'jwt'.
  • "password_confirmation does not match" → os dois campos de senha precisam ser idênticos.
  • Token expirado → use /api/refresh ou faça login de novo.
  • Postman continua 401 → verifique se o header é "Authorization" (com z!) e o valor começa com "Bearer " (com espaço).
